## 1. Обоснование идеи

В профессиональной среде знания поступают из множества источников:

* технические статьи
* посты в профессиональных каналах
* GitHub‑репозитории
* образовательные ресурсы

Большая часть этих материалов сохраняется вручную (закладки, избранное, заметки).

Такая форма хранения имеет ряд проблем:

* отсутствие структуры
* сложность поиска
* дублирование
* потеря контекста

В результате накопленные материалы редко используются повторно.

Агентная система может автоматизировать процесс **курирования знаний**:

* обрабатывать входящий поток контента
* извлекать ключевые идеи
* классифицировать материалы
* формировать структурированную базу знаний

Такой инструмент может использоваться как индивидуально, так и внутри команд.

---

## 2. Цель проекта

Создать PoC агентной системы, которая автоматически обрабатывает поток информационных ресурсов и превращает его в структурированную базу знаний.

---

## 3. Метрики успеха

### Продуктовые

* релевантность результатов поиска
* полезность подборок ресурсов

### Агентские

* точность классификации тем
* корректность извлечения тегов
* доля успешно обработанных источников

### Технические

* latency ответа системы
* стоимость обработки одного ресурса

---

## 4. Потенциальные сценарии использования

### Сценарий 1

Пользователь сохраняет пост из Telegram с полезной статьёй.

Система:

* извлекает ссылку
* парсит статью
* определяет тему
* сохраняет в базу знаний.

---

### Сценарий 2

Пользователь добавляет ссылку на GitHub‑репозиторий.

Система:

* извлекает описание проекта
* определяет технологический стек
* создаёт карточку ресурса.

---

### Сценарий 3

Пользователь делает запрос:

> "Покажи ресурсы по diffusion models"

Система возвращает:

* статьи
* репозитории
* обучающие материалы.

---

### Edge cases

* невалидная ссылка
* дублирующийся ресурс
* сайт с paywall
* слишком короткий пост
* контент без явной темы

---

## 5. Ограничения

### Технические

* p95 latency ответа: до 5 секунд
* обработка одного ресурса: до 20 секунд
* ограничение на размер входного текста

### Операционные

* ограниченный бюджет на LLM
* ограниченное количество источников данных

---

## 6. Архитектурный набросок

Система состоит из следующих компонентов:

### 1. Ingestion

Принимает:

* URL
* текст
* Telegram пост

---

### 2. Parsing

Извлекает текст и метаданные из источника.

---

### 3. Knowledge Processing

Использует LLM для:

* генерации краткого описания
* извлечения тегов
* определения темы

---

### 4. Storage

Сохраняет:

* текст
* метаданные
* embedding

---

### 5. Retrieval

Обрабатывает пользовательские запросы и возвращает релевантные ресурсы.

---

## 7. Потенциальный Data Flow

1. пользователь добавляет ресурс
2. система извлекает текст
3. происходит очистка данных
4. LLM извлекает теги и описание
5. данные сохраняются
6. пользователь делает запрос
7. retrieval модуль возвращает подборку
